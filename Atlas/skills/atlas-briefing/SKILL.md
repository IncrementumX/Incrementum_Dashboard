# atlas-briefing — daily news ingestion

Every morning before the CIO reads anything else, this skill runs and produces a dated briefing digest. It is the equivalent of the 7:15am newsprint-and-Bloomberg scan a portfolio manager does while drinking coffee.

## Inputs

### Standing target list (cross-reference everything against)

Loaded from `work_queues/briefing.md`. At time of writing:

- **Live positions:** BESI, HII, AGQ.
- **Active watchlist:** per `agents_context/state.md`.
- **Active themes:** AI capex / advanced packaging, US defense cycle, sovereign credit / reserve regime.

If `agents_context/state.md` changes, the target list updates automatically — do not hardcode tickers in this skill.

### Source set

English, institutional:
- WSJ (wsj.com, Heard on the Street column priority)
- Barron's (barrons.com)
- FT (ft.com)
- Bloomberg (if Eduardo has exported the terminal digest to `raw/briefings/bloomberg-<YYYY-MM-DD>.html`)
- Reuters (reuters.com)

Brazilian:
- Valor Econômico (valor.globo.com)
- Estadão (economia.estadao.com.br)
- Pipeline (pipelinevalor.globo.com)
- BrazilJournal (braziljournal.com)
- NeoFeed (neofeed.com.br)

IR / primary:
- Company IR pages for each live position + top-10 watchlist (for filings, press releases).

Sell-side (when Eduardo forwards):
- Any deck/note Eduardo drops into `raw/research/`.

## Process

### Phase 1 — pull

For each target ticker + theme, pull recent material from the source set. Preferred engine (long-term): Perplexity Agent API. Current reality:

- **If Perplexity is wired** (env var `PERPLEXITY_API_KEY` present): use the Perplexity pplx client with queries like `"<ticker> today earnings OR guidance OR litigation OR strategic"` across the source set.
- **If Perplexity is not wired**: fall back to Claude web_search with the same query structure, OR read manually-dropped files in `raw/briefings/inbox/<YYYY-MM-DD>/`.

Tag every pulled item with its `{src, pulled, cred}` metadata. Perplexity output is D-cred by default — validate key claims against the primary source before surfacing.

### Phase 2 — score

For each item, assign a priority:

- **CRITICAL** — could trigger a kill-shot rule on a live position. Route immediately to `work_queues/monitor.md` with `Severity: critical`, not just into the briefing digest.
- **HIGH** — material for a live position but not kill-shot-adjacent (guidance change, management turnover, major customer move).
- **MEDIUM** — thematic drift (competitor news, sector headline, macro data that updates a theme page).
- **LOW** — context / color, not actionable but worth a pointer.

Drop anything that doesn't score at least LOW. A 200-line briefing that no one reads is worse than a 20-line briefing the CIO reads every morning.

### Phase 3 — write

Output path: `wiki/summaries/<YYYY-MM-DD>-briefing.md`.

Structure:

```markdown

_Generated by atlas-briefing at <HH:MM America/Sao_Paulo>. Sources: <list>._

## TL;DR (≤ 5 bullets)
- <the 5 things the CIO needs to know before the market opens>

## Live positions
### BESI
- <item> [src: <url>, pulled: <date>, cred: <A|B|C|D>]
### HII
...
### AGQ
...

## Themes
### AI capex / advanced packaging
...
### US defense cycle
...
### Sovereign credit
...

## Watchlist deltas
- <if any watchlist name moved materially>

## Macro
- <CPI prints, central bank actions, rates moves, FX moves if material>

## Routed to monitor queue (CRITICAL / HIGH)
- <flag_id> — <ticker> — <one-line summary>
```

### Phase 4 — route

If any item scored CRITICAL or HIGH, also write it to `work_queues/monitor.md` with the proper schema (see `work_queues/monitor.md` header). The CIO triages the monitor queue separately from the briefing digest.

### Phase 5 — log

Append to `wiki/log.md`:
```
INGEST <YYYY-MM-DD HH:MM> <task_id> briefing-<YYYY-MM-DD> -> summaries/<YYYY-MM-DD>-briefing.md + monitor flags (N)
```

## Boundaries

1. **Never surface a claim without a source tag.** If the source can't be named, the claim doesn't run.
2. **Never mix D-cred Perplexity output into a position's main evidence.** D-cred stays in the briefing digest as "flagged, needs validation" unless cross-referenced to A/B.
3. **Never cut a TL;DR longer than 5 bullets.** If the CIO is scanning it at 7:15am, anything past bullet 5 doesn't get read.
4. **Never skip the macro section on days the US CPI, FOMC, Selic, or BCE decision prints.**
5. **Never run this skill on weekends unless Eduardo requests** — Saturday/Sunday is optional and low-signal.
